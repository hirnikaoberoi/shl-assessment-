# Approach

## Data: sourcing the catalog

The assignment points at `shl.com/solutions/products/product-catalog/`, restricted to
Individual Test Solutions. That URL now 301-redirects to a generic marketing page — SHL
restructured the site since the assignment was written (confirmed via `curl -IL`, not
assumed). The catalog is still fully intact in the Wayback Machine, so I used the CDX API
to resolve the freshest available snapshot per paginated listing page
(`?start=N&type=1`), then scraped each listing page and every assessment's detail page.

`type=1`/`type=2` cleanly separate **Individual Test Solutions** from **Pre-packaged Job
Solutions** in SHL's own markup. I scrape only `type=1`, and scope extraction to the
`<table>` whose header literally says "Individual Test Solutions" rather than trusting
item names — a name-based filter would have wrongly excluded legitimate items like "Entry
Level Sales Solution" (a composite personality/competency test that happens to be branded
"Solution" but is genuinely listed under Individual Test Solutions on SHL's own page).
Trusting the structural markup over a name heuristic was the right call.

Every catalog URL is a real, live `shl.com` link, verified to resolve during scraping.
Result: 361 Individual Test Solutions with name, description, job levels, test type(s),
duration, remote/adaptive support, and language, in `data/processed/catalog.json`.

## Retrieval: BM25, not embeddings

A from-scratch BM25 index over each assessment's concatenated text (name, description,
job levels, test types, remote/adaptive flags), plus a small hand-built synonym map
expanding recruiter vocabulary ("developer" → programmer/engineer/coding, "leadership" →
manager/executive/director) before scoring. No embedding model, vector DB, or external
embedding call.

Retrieval sits on the hot path of every `/chat` call under a 30-second budget — an
embedding API adds latency, cost, and a failure mode purely for retrieval, and a local
embedding model (sentence-transformers) drags in `torch`, a multi-hundred-MB dependency
that risks build time and memory on a free-tier host. BM25 is a few dozen lines of pure
Python: zero runtime dependency risk, fully unit-testable offline. The catalog is a few
hundred short, keyword-dense documents, not long-form prose, so lexical search is a
reasonable fit — the synonym layer covers where it would otherwise miss (role-title
variation, seniority phrasing).

## Conversation control flow: rule-based, not LLM-classified

Whether a turn should clarify, recommend, refine, compare, or refuse is decided by
deterministic Python (`classifier.py`), not an LLM self-classifying. The assignment warns
that a non-deterministic conversation shouldn't make the system fall apart — if control
flow itself depended on an LLM call, every downstream behavior would inherit its
variance. Rule-based classification means all four required behaviors are unit-testable
with no API key (`tests/test_classifier.py`, `tests/test_response.py`). The LLM is used
only for grounded text generation — clarifying questions, recommendation rationale,
comparisons — never for deciding *whether* to do those things.

**Out-of-scope and prompt-injection turns never reach the LLM at all** — caught by regex
and answered with a fixed template. A refusal generated by an LLM can in principle be
talked out of refusing; a hardcoded string cannot.

## Structured output over marker-parsing

An earlier iteration of this idea asked the LLM for free text with a
`RECOMMENDED_ASSESSMENTS:` bullet block, regex-extracted afterward. That broke silently:
the extractor scanned the *entire* reply for lines starting with `-`, so a comparison
answer's bullet points could get misread as recommendations, wrongly populating
`recommendations` and flipping `end_of_conversation` to true on a purely informational
turn.

This version asks Groq for strict JSON (`response_format: json_object`): `{"reply": ...,
"recommended_assessment_names": [...]}`. The real fix is structural, not prompt-based:
`response.py` only reads `recommended_assessment_names` when `query_type` is
`"recommendation"` or `"refinement"` — every other turn type hardcodes `recommendations`
to `[]` regardless of what the model returned. A prompt can be ignored by the model; an
`if` statement in the response builder cannot. Every returned name is then independently
re-resolved against the real catalog (exact match, falling back to a close-match search
restricted to the retrieved candidates only) before it can reach the API response, so a
hallucinated name is filtered out rather than surfaced.

## Statelessness

The API is stateless, but the agent still needs "what did I recommend last turn" to
refine, and "how many times have I asked" to bound clarification. Since the client only
echoes back `role`/`content` pairs, I encode a plain recap line into the `reply` text
itself (`"Shortlisted assessments: X | Y | Z"`) and parse it back out of history on the
next call — worded to read as an ordinary sentence, not a hidden token, so it's harmless
if the simulated user reads it back.

## Evaluation

Two layers, since I had neither the assignment's own traces nor a Groq key at build time:

1. **34 hermetic tests, no API key needed:** classifier correctness (vague → clarify,
   rich context → recommend, JD paste skips clarification, refinement requires a prior
   shortlist, injection/off-topic detection), BM25 ranking sanity, catalog scope-filter
   correctness, and — most importantly — `response.py`'s gating logic tested with a
   *mocked* LLM that deliberately misbehaves (returns names on a comparison turn, invents
   a nonexistent name) to prove the structural gates hold regardless of model behavior.
2. **`scripts/run_traces.py`:** 10 self-authored persona traces covering all four
   required behaviors plus refusal/injection probes, run end-to-end against the real
   Groq-backed pipeline (10/10 passing against live output). Requires `GROQ_API_KEY`;
   skips with a clear message instead of failing if unset, so `pytest` stays green
   everywhere.

## What I didn't get to

Recall@10 tuning against the assignment's own labeled traces — without them, synonym
coverage is informed by my own trace set, not ground truth. A short "closing" fast-path
(recognizing "thanks, that works" and replying without another LLM call) would also cut
redundant final-turn latency.
