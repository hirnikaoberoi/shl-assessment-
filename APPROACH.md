# Approach

## Data: sourcing the catalog

The assignment points at `shl.com/solutions/products/product-catalog/`, restricted to
Individual Test Solutions. That URL now 301-redirects to a generic marketing page — SHL
restructured the site since the assignment was written. I confirmed this by fetching it
directly (`curl -IL`) rather than assuming, then found the catalog is still fully intact
in the Wayback Machine, which has been crawling it since 2021. I used the CDX API to
resolve, per paginated listing page (`?start=N&type=1`), the freshest available snapshot,
then scraped each page and every individual assessment's detail page. `type=1` /
`type=2` cleanly separate **Individual Test Solutions** from **Pre-packaged Job
Solutions** in SHL's own markup — I scrape only `type=1`, and additionally scope
extraction to the `<table>` whose header literally says "Individual Test Solutions" (not
just "whichever table has rows"), so a Job Solution can never enter the catalog. I
deliberately used this structural signal instead of a name-based heuristic (e.g.
flagging anything named "...Solution"): I initially suspected items like "Entry Level
Sales Solution" were miscategorized Job Solutions based on their name, but the
type-scoped table extraction confirms SHL itself lists them under Individual Test
Solutions (they're composite personality/competency assessments that happen to be
branded "Solution") — so a name heuristic would have wrongly excluded legitimate
catalog items. Trusting SHL's own structural markup over pattern-matching on the name
was the right call here.

Every catalog URL is a real, live `shl.com` link (old catalog paths 301-redirect to
SHL's current page structure, verified during scraping) — satisfying "every URL must
come from your scraped catalog" with URLs that actually resolve today. Result: 361
Individual Test Solutions with name, description, job levels, test type(s), duration,
remote/adaptive support, and language, stored in `data/processed/catalog.json`.

## Retrieval: BM25, not embeddings

I used a from-scratch BM25 index over each assessment's concatenated text (name,
description, job levels, test types, remote/adaptive flags), with a small hand-built
synonym map that expands recruiter vocabulary ("developer" → programmer/engineer/coding,
"leadership" → manager/executive/director, etc.) before scoring. No embedding model, no
vector DB, no external embedding API call.

Why: retrieval sits on the hot path of every `/chat` call under a 30-second budget.
Calling an embedding API adds latency, cost, and a failure mode (timeout, rate limit)
purely for retrieval. A local embedding model (sentence-transformers) is deterministic
and fast but drags in `torch` — a multi-hundred-MB dependency that materially risks
build time and memory on a free-tier host. BM25 is a few dozen lines of pure Python,
has zero runtime dependency risk, and is fully unit-testable without any network or GPU.
Given the catalog is a few hundred short, keyword-dense documents (not long-form prose),
lexical search is a reasonable fit for its cost — the synonym layer specifically targets
where lexical search would otherwise miss (role-title variation, seniority phrasing).

## Conversation control flow: rule-based, not LLM-classified

Whether a turn should clarify, recommend, refine, compare, or refuse is decided by
deterministic Python (`classifier.py`), not by asking the LLM to self-classify. The
assignment explicitly warns that "a non-deterministic conversation should not make the
system fall apart" — if the *control flow* itself depends on an LLM call, then every
piece of downstream behavior (whether recommendations get populated, whether the
conversation ends) inherits the LLM's variance. Keeping classification rule-based means
the four required behaviors are testable with plain unit tests and no API key (see
`tests/test_classifier.py`, `tests/test_response.py` — 29 tests, all hermetic).

The LLM is used only for grounded text generation: phrasing clarifying questions,
writing the recommendation rationale, and comparing named assessments — never for
deciding *whether* to do those things.

**Out-of-scope and prompt-injection turns never reach the LLM at all.** They're caught
by regex (`classifier.py::is_out_of_scope`) and answered with a fixed template. This is
a deliberate defense-in-depth choice: a refusal that's generated by an LLM can in
principle be talked out of refusing; a refusal that's a hardcoded string cannot.

## Structured output over marker-parsing

The previous iteration of this idea asked the LLM to emit free text with a
`RECOMMENDED_ASSESSMENTS:` bullet block, then regex-extracted it. That broke silently:
the extractor scanned the *entire* reply for lines starting with `-`, so a comparison
answer's bullet points (`- Purpose: ...`, `- Focus: ...`) could get misread as
recommendations, populating `recommendations` and flipping `end_of_conversation` to true
on a turn that was supposed to be purely informational.

This version asks Groq for a strict JSON object (`response_format: json_object`):
`{"reply": ..., "recommended_assessment_names": [...]}`. More importantly, the gate is
structural, not prompt-based: `response.py` only ever looks at
`recommended_assessment_names` when `query_type` is `"recommendation"` or
`"refinement"` — for every other turn type, `recommendations` is hardcoded to `[]`
regardless of what the model returned. A prompt can be ignored by the model; an `if`
statement in the response builder cannot. Every name the model does return is then
independently re-resolved against the real catalog (exact match, falling back to a
close-match search restricted to the retrieved candidate pool only) before it can appear
in the API response — a hallucinated name is filtered out, never surfaced.

## Statelessness

The API is stateless per the spec, but the agent still needs to know "what did I
recommend last turn" to refine, and "how many times have I already asked" to bound
clarification. Since the client only echoes back `role`/`content` pairs, I encode a
plain, human-readable recap line into the `reply` text itself
(`"Shortlisted assessments: X | Y | Z"`) and parse it back out of the message history on
the next call. It's deliberately worded to read as an ordinary sentence, not a hidden
token, so it's harmless if the eval harness's simulated user reads it back.

## Evaluation

Two layers, since I didn't have the assignment's own 10 traces or a Groq API key at
build time:

1. **Hermetic tests (29, all passing, no API key needed):** classifier correctness
   (vague → clarify, rich context → recommend, JD paste skips clarification, refinement
   requires a prior shortlist, injection/off-topic detection), BM25 ranking sanity, and —
   most importantly — `response.py`'s gating logic tested with a *mocked* LLM that
   deliberately misbehaves (returns names on a comparison turn, invents a nonexistent
   assessment name) to prove the structural gates hold regardless of what the model does.
2. **`scripts/run_traces.py`:** 10 self-authored persona traces
   (`tests/conversation_traces/`) covering all four required behaviors plus refusal and
   injection probes, run end-to-end against the real Groq-backed pipeline. Each trace
   asserts the classified `query_type` per turn and the final recommendation count.
   Requires `GROQ_API_KEY`; the harness prints a clear message and skips instead of
   failing if it's unset, so `pytest` stays green in any environment.

## What I didn't get to / would do next

- Recall@10 tuning against the assignment's own labeled traces — I don't have them, so
  synonym coverage is informed by my own trace set, not ground truth. I'd expect to
  extend the synonym map and possibly add test-type-aware query boosting once I can
  measure against real labels.
- A short "closing" fast-path (recognizing "thanks, that works" after a shortlist and
  responding without another LLM call) would reduce redundant final-turn latency.
